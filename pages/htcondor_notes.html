01 What is HTCondor?

Core concept ¬∑ High-Throughput Computing system developed at UW‚ÄìMadison

Definition

HTCondor is an open-source software system that creates a High-Throughput Computing (HTC) environment. It effectively uses the computing power of machines connected over a network ‚Äî be they a single cluster, a set of campus clusters, standalone cloud resources, or international grids. Its power comes from the ability to harness shared resources with distributed ownership.

üîë Key Insight ‚Äî HTC vs HPC

High-Performance Computing (HPC) delivers enormous power over a short time (measured in FLOPS per second). High-Throughput Computing (HTC) delivers large amounts of power over a long time ‚Äî scientists care about floating-point operations per month or year, not per second. HTCondor is built specifically for this model.

Origin

Developed by the HTCondor team at the University of Wisconsin‚ÄìMadison, Computer Sciences Department. Free and open-source under Apache License 2.0.

‚öôÔ∏è
Platform Support

Runs on Linux, Unix, macOS, FreeBSD, and Windows. Can integrate dedicated rack clusters and idle desktop machines (cycle scavenging) into one pool.

üîÅ
Job Checkpointing

Jobs can produce and resume from checkpoints ‚Äî saving weeks of computation if a machine crashes or is rebooted for maintenance.

üéØ
ClassAds Matchmaking

Machines advertise their resources (RAM, CPU speed, load). Jobs submit requirements. HTCondor matches them ‚Äî like classified newspaper ads. No need to pick queues manually.

Key Features at a Glance

  • No account (login) required on remote worker machines ‚Äî HTCondor uses file transfer and split execution mechanisms.
  • Supports DAGMan for workflow dependencies: job C starts only after jobs A and B complete.
  • Supports sequential and parallel jobs across multiple "universes" (vanilla, standard, local, MPI, PVM).
  • Idle desktops are put to work; jobs are preempted if the owner returns (keystroke, mouse move, high CPU usage).
  • HTCondor-G allows jobs to use resources outside its direct control: Globus, Nordugrid ARC, UNICORE, Amazon EC2, PBS/Torque, LSF, and more.
  • GlideinWMS / Glidein makes remote Globus grid resources transparently part of a virtual HTCondor cluster.

Core Concepts Worth Mastering

  • ClassAds language ‚Äî The declarative matchmaking system that describes both job requirements and machine capabilities.
  • Universes ‚Äî Execution environments for jobs: vanilla (most programs), standard (checkpointing + remote I/O), docker, MPI, local, and more.

03 HTCondor-CE ‚Äî Compute Entrypoint

Grid gateway ¬∑ The "door" to your local cluster for the outside world

What is HTCondor-CE?

HTCondor-CE (Compute Entrypoint) is a special configuration of HTCondor designed as a gateway for computing grids such as the European Grid Infrastructure (EGI) and the Open Science Grid (OSG). It acts as the "door" through which remote organisations submit Resource Allocation Requests (RARs), which are then authorised and delegated to the site's local batch system.

üìå In Plain English

Imagine your institution has a cluster running Slurm. You want to join a global research grid so external scientists can run jobs on your machines. HTCondor-CE sits at the edge of your network, accepts incoming job requests from the grid, checks they are authorised, transforms them into Slurm jobs, and submits them ‚Äî all automatically. Your existing Slurm setup doesn't change.

üîê
Security

Uses host certificates (PKI/CAs) and bearer tokens for authentication. The CE presents certificates to prove its identity, like HTTPS for websites.

üîÄ
Job Routing

The Job Router daemon transforms incoming grid jobs into local batch jobs via configuration variables ‚Äî no code modification needed.

üõ©Ô∏è
Pilot Jobs

Most incoming grid jobs are pilot jobs from factories (e.g. GlideinWMS). They create containers/environments for actual end-user jobs to run inside.

‚ö°
Scale

Responsible for provisioning ~9 million core-hours per day across ~170 endpoints globally, as part of the OSG Compute Federation.

Why Use HTCondor-CE?

  • Grid participation without rewriting your batch system ‚Äî CE is a thin layer on top of your existing Slurm, PBS, LSF, or HTCondor installation.
  • Supports all major batch systems ‚Äî Grid Engine, HTCondor, LSF, PBS Pro/Torque, and Slurm are all natively supported.
  • Routing as configuration ‚Äî Job transformation rules are defined in config files, so they persist across software upgrades without touching internals.
  • Standard security model ‚Äî Integrates with the established OSG/EGI PKI and token-based auth infrastructure.
  • Open access for science ‚Äî Enables your resources to serve international research communities (LHC experiments, bioinformatics, astrophysics, etc.).
  • Developed & maintained by CHTC ‚Äî The Center for High Throughput Computing at UW‚ÄìMadison provides ongoing support and releases.
